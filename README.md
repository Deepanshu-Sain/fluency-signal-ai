# The Fluency Signal

An AI collaboration assessment tool that measures **how effectively users work with AI during a realistic writing task**.

The goal is not to evaluate the final email itself, but to understand the user's **AI collaboration behaviour** — how they prompt, verify information, iterate on drafts, and make independent decisions while working with an AI assistant.

Built as a full-stack Next.js application combining:

- Real-time AI chat interaction
- Persistent session tracking
- Draft revision analysis
- Behavioural signal extraction
- LLM-generated evaluation feedback


## Overview

The Fluency Signal simulates a real-world workflow:

1. A user receives a cold outreach writing task.
2. They collaborate with an AI writing assistant.
3. They refine their email draft through multiple iterations.
4. Their interactions are stored and analysed.
5. A report is generated showing measurable AI collaboration signals.

The system evaluates three primary signals:

| Signal | Purpose |
|---|---|
| Prompt Count | Measures AI engagement frequency |
| Edit Ratio | Measures how much the draft evolved through revisions |
| Verification Signal | Detects attempts to validate, challenge, or verify AI outputs |

## Flow Explanation

### 1. Task Completion

The user receives a realistic cold outreach task and works on the email using two parallel surfaces:

- **AI Assistant**
  - Used for brainstorming, refinement, feedback, personalization, and challenging assumptions.

- **Email Editor**
  - Used for drafting, editing, and revising the final email response.

---

### 2. Interaction Capture

Every meaningful interaction during the session is persisted in the database:

- User and assistant chat messages are stored with timestamps.
- Email draft versions are captured throughout the editing process.
- Meaningful edit events are recorded by comparing changes between draft versions over time.

---

### 3. Signal Extraction

After submission, the application analyzes the complete session and calculates measurable AI collaboration signals.

#### Prompt Count

Measures how frequently the user engaged with the AI assistant during the task.

#### Verification Signal

Detects whether the user attempted to validate information, question assumptions, request fact-checking, or review AI-generated suggestions.

#### Edit Ratio

Measures how much the email draft evolved through revision history instead of being accepted unchanged from the initial version.

---

### 4. AI-Based Evaluation

A second Gemini evaluation call analyzes the complete session evidence:

- Conversation history
- Final email draft
- Computed collaboration signals

The evaluator generates an evidence-based interpretation of the user's AI collaboration behaviour, focusing on iteration quality, verification behaviour, feedback usage, and independent decision-making.

---

### 5. Assessment Report

The user receives a final assessment report containing:

- Collaboration metrics
- Overall collaboration score
- Behavioural strengths
- Suggested improvements
- AI-generated interpretation of the session

# Signal Definitions

## 1. Prompt Count

### Definition

Number of messages sent by the user to the AI assistant.


### Calculation

```text
Prompt Count =
COUNT(chat_messages WHERE role = "user")
```


### Reasoning

Prompt count measures AI engagement frequency.

A higher number does not automatically mean better collaboration, but it indicates that the user actively involved AI during the workflow.


---

# 2. Verification Signal

### Definition

Detects whether the user attempted to verify, challenge, or validate AI-generated information.


### Detection

Verification behaviour is detected using pattern matching on user messages.

Examples include:

- "Can you verify this?"
- "Is this accurate?"
- "What is the source?"
- "Are we assuming this?"
- "Find flaws in this idea"


### Calculation

```text
Verification Signals =
COUNT(chat_messages WHERE verification_signal = 1)
```


### Reasoning

Effective AI collaboration requires judgment.

The system rewards users who treat AI output critically rather than blindly accepting generated information.


Important:

Verification signals measure **attempts to verify**, not whether the verification itself was correct.


---

# 3. Edit Ratio

### Definition

Measures how much the email draft changed throughout the revision process.


### Meaningful Edit Definition

An edit is considered meaningful when:

```text
characters_changed >= 50

AND

edit_ratio_contribution >= 0.05
```


Small formatting changes, cursor movement, and minor edits are ignored.


### Formula

For every meaningful revision:

```text
edit_ratio_contribution =
characters_changed /
previous_draft_length
```


The final edit ratio is:

```text
Edit Ratio =
MIN(
SUM(edit_ratio_contribution),
1
)
```


Example:

```
Previous draft length: 1000 characters

User changes: 300 characters


Contribution:

300 / 1000 = 0.3


Edit Ratio = 30%
```


### Reasoning

The goal is not to judge whether edits improved the email.

The signal measures transformation activity — whether the user actively shaped the draft instead of simply accepting initial output.

# Scoring Methodology


## Collaboration Score

The collaboration score combines:

| Component | Weight |
|---|---:|
| Prompt Engagement | 25 |
| Verification Behaviour | 25 |
| Interaction Depth | 50 |


Formula:

```
Collaboration Score =
Prompt Score
+
Verification Score
+
Interaction Score
```


Maximum:

```
100
```


---

## Iteration Score

Measures draft evolution.

Components:

| Component | Weight |
|---|---:|
| Edit Ratio | 80 |
| Meaningful Edit Presence | 20 |


Formula:

```
Iteration Score =
(Edit Ratio × 80)
+
Meaningful Edit Score
```


The edit ratio contribution is capped to avoid excessive scoring from repeated modifications.


---

## Overall Score

Final score:

```
Overall Score =
(Collaboration Score × 0.5)
+
(Iteration Score × 0.5)
```


This balances:

- How the user worked with AI
- How the user's draft evolved


---

# LLM Evaluation Layer

After calculating numerical signals, a second Gemini evaluation call generates qualitative feedback.


The evaluator receives:

- Session signals
- Complete conversation history
- Final draft


The evaluator focuses only on:

- AI usage behaviour
- Feedback seeking behaviour
- Verification behaviour
- Revision behaviour
- Independent decision making


It intentionally avoids judging:

- Email quality
- Grammar
- Business effectiveness


The evaluation follows an evidence-first approach:

- Explicit user behaviour is strongest evidence.
- AI conversation context is secondary evidence.
- Final draft differences alone are weak evidence.

# Design Decisions & Trade-offs


## Why SQLite?

SQLite was selected because:

- The application is single-user.
- No deployment database infrastructure is required.
- Local persistence is sufficient for the assessment workflow.


A production version would likely migrate to:

- PostgreSQL
- Supabase
- Managed database infrastructure


---

## Why heuristic verification detection?

Verification behaviour is detected using lightweight pattern matching.

Advantages:

- Simple
- Explainable
- Fast


A production implementation could combine:

- LLM classification
- Semantic similarity
- Behaviour embeddings


---

## Future Improvements

With more time, improvements would include:

- Better semantic verification detection
- More advanced edit attribution
- User authentication
- Multi-user sessions
- Production database
- Analytics dashboard
- More behavioural signals
